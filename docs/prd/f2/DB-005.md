# DB-005 : Tables notifications — templates par banque, opt-in/consent, journal d'envoi, devices push

**Module** : F2 — Data · **Agent** : agent-database · **Dépend de** : DB-004, CONTRACT-007 · **Statut** : TODO

## Exigences (EARS)

- `notification_templates` : `bank_id`, `type` (enum NotificationType de LA LOI), `channel` (enum NotificationChannel), `lang` (fr/dioula/baoule/en), `body` text — unique `(bank_id, type, channel, lang)` ; les variables autorisées (`{{number}}`, `{{position}}`, `{{estimate}}`) sont validées côté API (CONTRACT-005), la base stocke.
- `notification_consents` : `bank_id`, `phone_encrypted`, **`phone_hash` (HMAC-SHA256 clé env, colonne de recherche — le téléphone en clair n'est JAMAIS indexable)**, `channel`, `opted_in` boolean, `opted_at`, `revoked_at?` — unique `(bank_id, phone_hash, channel)`.
- `notification_log` : `bank_id`, `ticket_id?`, `type`, `channel`, `phone_hash?`, `device_id?`, `status` (QUEUED/SENT/DELIVERED/FAILED), `failure_reason?` (enum de LA LOI), `provider_message_id?`, horodatages — index `(bank_id, ticket_id)` et `(bank_id, status, created_at)`. Le masquage d'affichage (`phoneNumberMasked`) est calculé par l'API, la base ne stocke jamais de forme masquée.
- `notification_devices` : `bank_id`, `device_token` (unique), `platform` (IOS/ANDROID/EXPO), `phone_hash?`, `last_seen`, `revoked_at?` — support POST/DELETE devices (CONTRACT-007), ré-enregistrement idempotent par token.
- `notification_test_recipients` (amendé post-critique — support 422 TEST_RECIPIENT_NOT_ALLOWED de LA LOI) : `bank_id`, `phone_hash`, `phone_encrypted`, `added_by`, `added_at` — unique `(bank_id, phone_hash)`.
- UBIQUITAIRE : aucune colonne téléphone en clair dans ces 5 tables ; type `phone_encrypted text` DÉFINITIF dès création (format DB-008, aucune ALTER ultérieure) ; RLS + cas tenant-isolation pour chacune.
