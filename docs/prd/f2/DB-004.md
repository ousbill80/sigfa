# DB-004 : Table audit_log immuable + triggers d'écriture

**Module** : F2 — Data · **Agent** : agent-database · **Dépend de** : DB-003 · **Statut** : TODO

## Exigences (EARS)

- La table `audit_log` doit porter : `id`, `bank_id`, `actor_id`, `actor_role`, **`actor_email text?`** (dénormalisé — l'acteur peut être supprimé), `action varchar(500)` (libre, style « PATCH /banks/:id/theme »), `entity_type`, `entity_id`, `occurred_at timestamptz DEFAULT now()`, `ip inet`, `diff` jsonb — **mapping DB→API documenté** (`occurred_at` → `timestamp`, actor composé) ; index `(bank_id, occurred_at)` et `(bank_id, entity_type, entity_id)`.
- **Immuabilité au niveau base** : UPDATE et DELETE sur audit_log doivent être IMPOSSIBLES même pour le rôle applicatif — triggers `BEFORE UPDATE/DELETE ... RAISE EXCEPTION` + REVOKE ; testé.
- Des **triggers d'audit** doivent journaliser automatiquement INSERT/UPDATE/DELETE des tables sensibles (banks, agencies, services, counters, users, kiosks — la liste est versionnée) avec `diff` = anciennes/nouvelles valeurs, EN EXCLUANT les colonnes sensibles **par motif de nom** (`*_hash`, `*_encrypted`, `*_cipher` — robuste aux colonnes futures, testé avec une colonne factice). UBIQUITAIRE : la table `tickets` est INTENTIONNELLEMENT exclue des triggers (fréquence d'UPDATE incompatible avec un trigger synchrone) — ses mutations sont journalisées applicativement par SEC-001 via `insertAuditEntry()` ; cette exclusion est verrouillée par test.
- Les mutations applicatives non-DDL (ex. transitions de ticket) seront journalisées par l'API (SEC-001) via la même table — la story expose `insertAuditEntry()` typé.
- RLS : lecture `bank_id = current_setting(...)` (l'écran Auditor est scopé banque) ; la purge à 24 mois est un job d'exploitation documenté (`src/rls/audit-retention.md`), PAS un trigger (immuabilité ≠ rétention — décision consignée).
- L'horloge doit être `occurred_at timestamptz NOT NULL DEFAULT now()` — jamais fournie par le client.
