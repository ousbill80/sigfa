# DB-008 : Chiffrement AES-256 des téléphones au repos + purge automatique 13 mois (droit à l'oubli UEMOA)

**Module** : F2 — Data · **Agent** : agent-database · **Dépend de** : DB-006 · **Statut** : TODO

## Exigences (EARS)

- Le package doit fournir un module `src/crypto/phone-cipher.ts` : chiffrement **AES-256-GCM applicatif** (clé 32 octets depuis `PHONE_ENCRYPTION_KEY` env — jamais en dur, `.env.example` documenté ; IV aléatoire par valeur ; format stocké `v1:iv:tag:ciphertext` versionné pour rotation future) + `hashPhone(raw)` HMAC-SHA256 (clé `PHONE_HASH_KEY`) qui **NORMALISE d'abord** (strip espaces/tirets, validation `^\+[1-9]\d{7,14}$`, sinon `InvalidPhoneError`) — normalisation centralisée, utilisés par toutes les écritures de `phone_encrypted text`/`phone_hash text` (tickets, users, consents, log, devices, test_recipients — colonnes créées au type définitif par DB-001/005, AUCUNE ALTER ici).
- SI `PHONE_ENCRYPTION_KEY` est absente ou de mauvaise taille, ALORS le module doit échouer AU DÉMARRAGE avec message explicite (jamais de fallback clair).
- Décision consignée : chiffrement applicatif (pas pgcrypto) — la clé ne transite jamais vers PostgreSQL, les dumps sont chiffrés par construction.
- **Purge droit à l'oubli** : `retention_policies` (`bank_id`, `phone_retention_months` défaut 13, bornes 1–60) + fonction `purgeExpiredPhones()` : QUAND un ticket est clos depuis plus de `retention_months`, ses `phone_encrypted`/`phone_hash` doivent être mis à NULL (le ticket agrégé demeure — anonymisation, pas suppression) ; idem consents révoqués expirés. Rejouable idempotente ; l'orchestration cron = F6/exploitation.
- **Purge manuelle** (support `POST /data/purge-phone` de LA LOI) : `purgePhone(bankId, phone)` → anonymise TOUTES les occurrences du téléphone chez ce tenant, retourne `{ purged, affectedTickets }` idempotent (2e appel → purged:false).
- La purge doit écrire une entrée `audit_log` (action DATA_PURGE, sans le téléphone en clair — le hash tronqué suffit).
