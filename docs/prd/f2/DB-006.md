# DB-006 : Tables reporting — agrégats journaliers matérialisés

**Module** : F2 — Data · **Agent** : agent-database · **Dépend de** : DB-005, CONTRACT-006 · **Statut** : TODO

## Exigences (EARS)

- `daily_agency_stats` : `bank_id`, `agency_id`, `service_id?` (null = toutes), `day` date (fuseau Africa/Abidjan), et les mesures nécessaires aux 7 KPIs de LA LOI : `tickets_issued/served/abandoned/no_show`, `total_wait_seconds`, `total_service_seconds`, `sla_met_count`, `sla_total_count`, `feedback_count`, `feedback_sum`, `nps_promoters/passives/detractors`, `agent_active_seconds?` (**source : agrégation d'`agent_status_history` de DB-001**) — unicité par **deux index uniques partiels** : `(bank_id, agency_id, day) WHERE service_id IS NULL` et `(bank_id, agency_id, service_id, day) WHERE service_id IS NOT NULL` (Drizzle : `uniqueIndex(...).where(isNull(...))` / `.where(isNotNull(...))`).
- Les KPIs (TMA/TMT/TTS/abandon/taux SLA/NPS/occupation) sont des **calculs dérivés** de ces mesures — les formules exactes vivent dans REP-001 (API) ; la story garantit que chaque KPI de LA LOI est CALCULABLE depuis les colonnes (test de suffisance documenté champ→KPI).
- La table doit être remplissable de façon **idempotente et incrémentale** : fonction `upsertDailyStats(day, agencyId)` recalculant depuis `tickets` (source de vérité) — rejouable sans doublon ; l'orchestration (cron) est REP-001.
- Index : `(bank_id, day)` et `(bank_id, agency_id, day)` — bank_id en tête.
- `export_jobs` : `bank_id`, `requested_by`, `scope`, `period`, `format` (pdf/xlsx/json), `status` (PENDING/PROCESSING/READY/FAILED — enum LA LOI), `file_url?`, `expires_at?` — support du contrat d'export asynchrone.
- UBIQUITAIRE : aucun champ personnel dans daily_agency_stats (agrégats purs — conformité AnonymizedNetworkAggregate).
