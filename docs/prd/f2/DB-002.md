# DB-002 : RLS sur toutes les tables + contexte tenant SQL + suite tenant-isolation initiale

**Module** : F2 — Data · **Agent** : agent-database (périmètre étendu : + `packages/testing/src/tenant-isolation/`) · **Dépend de** : DB-001 · **Statut** : TODO

## Exigences (EARS)

- CHAQUE table portant `bank_id` doit avoir `ENABLE ROW LEVEL SECURITY` + `FORCE ROW LEVEL SECURITY` et une policy `tenant_isolation` : `USING (bank_id = current_setting('app.current_bank_id')::uuid)` — fichiers SQL versionnés dans `src/rls/`, appliqués par migration.
- Le package doit exposer un helper typé `withTenant(db, bankId, fn)` qui exécute `SET LOCAL app.current_bank_id` dans une transaction — l'API (F3) l'utilisera ; JAMAIS de requête métier hors contexte tenant.
- SI `app.current_bank_id` n'est pas positionné, ALORS toute requête sur une table protégée doit retourner zéro ligne (et les écritures échouer) — comportement testé.
- **Double rôle SQL (amendé post-critique)** : rôle **migrateur** (owner, utilisé par drizzle-kit) et rôle **applicatif** `sigfa_app` (non-owner, GRANT CRUD uniquement, PAS BYPASSRLS) — provisioning versionné `src/rls/roles.sql`, appliqué en migration ; le harness Testcontainers est étendu (périmètre consigné) pour retourner `migrationConnectionString` ET `appConnectionString` ; TOUS les tests RLS utilisent la connexion applicative (l'owner contourne FORCE RLS silencieusement).
- La **suite tenant-isolation** (T6) doit couvrir CHAQUE table de DB-001 : données banque A + banque B créées, contexte A → zéro ligne de B (SELECT/UPDATE/DELETE), tentative d'écriture avec `bank_id` de B sous contexte A → rejetée (policy WITH CHECK).
- QUAND une future story ajoute une table, le harness doit offrir un utilitaire générique `assertTenantIsolated(table, fixtures)` réutilisable (généralisation dans `@sigfa/testing`).
