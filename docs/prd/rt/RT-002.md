## Exigences (EARS)
- **UBIQUITAIRE** : suite `realtime-guarantees` bout-en-bout sur sockets réels — protocole de mesure NORMÉ (D8) : `t0 = io.emit` serveur, `t1 = handler `ticket:called`` client, **≥50 échantillons, warm-up exclu, même hôte (loopback + PG/Redis Testcontainers), transport websocket, adapter Redis ACTIF**. Seuil **p95 <500 ms** (`TICKET_CALLED_SLA_MS`) = SLA réseau loopback+adapter avec marge/répétitions ; en cas de flake CI → quarantaine/retry documenté, jamais un gate horloge-murale par appel.
- **QUAND** un client perd la connexion WS puis se reconnecte, **il resynchronise l'état** via `sync:request`→`sync:state` = **convergence d'ÉTAT** (snapshot : files, guichets, `recentCalls` TV ; `estimate` HORS scope — cf. arbitrage D5) : l'état FINAL client == état serveur, pas de rejeu d'événements manqués — testé par coupure/reprise réelle.
- **QUAND** 2 agents appellent le suivant simultanément (course réelle sur sockets + verrou Redis SET NX PX + `FOR UPDATE SKIP LOCKED`), **un seul obtient le ticket**, l'autre reçoit le suivant (pas d'erreur, pas de double `ticket:called` pour le même ticket) — vérifié de bout en bout (pas seulement au niveau service).
- **QUAND** l'adapter Redis pub/sub relaie entre 2 instances serveur, **un client abonné sur l'instance A reçoit un événement émis par l'instance B** (test multi-instance de l'adapter `@socket.io/redis-adapter`).
- **anormal** : si un payload d'émission est invalide, il n'est PAS diffusé (émetteur typé) et n'impacte pas la latence des émissions valides.

- [ ] `RT-002: coupure WS puis reconnexion → sync:state → convergence d'état (files/guichets/recentCalls exacts, estimate hors scope), pas de rejeu`
