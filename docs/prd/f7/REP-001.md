**Module** : F7 — Reporting · **Agent** : agent-api · **Dépend de** : DB-006, API-003 · **Statut** : TODO

Cette story livre les **formules EXACTES** des 7 KPIs de LA LOI (CONTRACT-006), la fonction d'orchestration incrémentale des agrégats, et la suite de tests d'exactitude **`sla-engine`**. Elle NE génère aucun PDF/Excel, n'envoie aucun mail, ne dessine aucun dashboard.

## Définitions métier EXACTES (fenêtre = jour civil `Africa/Abidjan`, UTC+0 sans DST)

Toutes les mesures s'appuient sur les colonnes matérialisées de `daily_agency_stats` (DB-006), source de vérité = `tickets`. Un ticket est **rattaché à un jour** par la date civile Abidjan de son `issued_at` (émission), pas de sa clôture. Un ticket ouvert la nuit et clos le lendemain reste comptabilisé au jour de son **émission** (voir cas multi-jours ci-dessous).

| KPI | Formule EXACTE | Population (numérateur / dénominateur) | Unité | Null si |
|---|---|---|---|---|
| **TMA** (Temps Moyen d'Attente) | `total_wait_seconds / served_count` | somme des `wait_seconds = called_at − issued_at` sur les tickets **appelés** ; dénominateur = tickets ayant reçu un 1er appel (`served_count`, statut ∈ {SERVING, DONE, NO_SHOW post-appel}) | secondes (entier, arrondi au plus proche) | `served_count = 0` |
| **TMT** (Temps Moyen de Traitement) | `total_service_seconds / done_count` | `service_seconds = closed_at − serving_started_at`, tickets **DONE** uniquement | secondes | `done_count = 0` |
| **TTS** (Temps Total de Service) | `TMA + TMT` | dérivé — null si l'un des deux est null | secondes | `TMA` ou `TMT` null |
| **Taux d'abandon** | `abandoned_count / (abandoned_count + served_count)` × 100 | ABANDONED = quitté la file **avant 1er appel** ; NO_SHOW (appelé mais absent) **exclu du numérateur** ; dénominateur = tickets ayant atteint un état terminal ou appelé | % (2 décimales) | dénominateur = 0 |
| **Taux SLA** | `sla_met_count / sla_total_count` × 100 | un ticket **respecte le SLA** si `wait_seconds ≤ SLA_service` (SLA configuré par service, DB-003) ; `sla_total_count` = tickets appelés + abandonnés (un abandon compte comme SLA **non respecté**) | % (2 décimales) | `sla_total_count = 0` |
| **NPS** | `((promoters − detractors) / feedback_count)` × 100 | promoter = note 5, passive = note 4, detractor ≤ 3 (mapping API-010) | score entier signé [−100..+100] | `feedback_count = 0` |
| **Occupation** | `agent_active_seconds / agent_available_seconds` × 100, plafonné à 100 | `agent_active_seconds` (temps ticket ouvert, DB-006, source `agent_status_history`) / temps où au moins un agent était en statut disponible/en service | % (2 décimales) | `agent_available_seconds = 0` |

## Exigences (EARS)

- **UBIQUITAIRE** — Chaque KPI est calculé par une fonction **pure** de `sla-engine` prenant un `DailyAgencyStats` (ou un agrégat multi-jours pré-sommé) et retournant `number | null`, sans accès DB, réseau, ni horloge — 100% déterministe.
- **UBIQUITAIRE** — Les 7 formules ci-dessus font LA LOI ; toute divergence entre le glossaire v5 §5 et cette table est tranchée par cette table (à faire valider — voir `_notes.md`).
- **UBIQUITAIRE** — Le rattachement d'un ticket à un jour utilise le **fuseau `Africa/Abidjan`** (UTC+00, pas de changement d'heure) : `day = date_civile_Abidjan(issued_at)`. La conversion est centralisée (`toAbidjanDay()`), jamais recalculée ad hoc.
- **ÉVÉNEMENT** — QUAND `GET /reports/kpis?scope=agency|network&period=` est appelé (rôle DIRECTOR+/AUDITOR, tenant scellé), le système agrège les `daily_agency_stats` de la période et retourne les 7 KPIs typés CONTRACT-006, chaque valeur avec son unité et sa nullabilité (NPS/occupation `null` documentés, jamais `0` fallacieux).
- **ÉVÉNEMENT** — QUAND `GET /reports/daily/:agencyId?day=` est appelé, le système retourne la structure du rapport journalier de LA LOI, dérivée des mesures du jour Abidjan demandé.
- **ÉVÉNEMENT** — QUAND l'agrégat d'un jour est (re)calculé, le système appelle `upsertDailyStats(day, agencyId)` (DB-006) — recalcul **idempotent** depuis `tickets`, rejouable sans doublon ; le moteur ne matérialise jamais un jour partiel comme définitif (voir cas multi-jours).
- **ÉTAT** — TANT QU'un jour Abidjan n'est pas clos (jour courant), les KPIs de ce jour sont marqués `partial: true` dans la réponse (agrégat en cours), jamais présentés comme figés.
- **ÉTAT** — TANT QU'une agrégation multi-jours est demandée, les moyennes sont recalculées à partir des **sommes** (`total_wait_seconds`, `served_count`…), jamais comme une « moyenne de moyennes » (interdit — biais).
- **INDÉSIRABLE (file vide / 0 observation)** — SI le dénominateur d'un KPI est nul (file vide, aucun ticket servi, aucun feedback), ALORS la fonction retourne `null` (jamais `0`, jamais `NaN`, jamais division par zéro).
- **INDÉSIRABLE (multi-jours)** — SI un ticket est émis un jour J et clos J+1, ALORS ses durées (`wait`, `service`) sont attribuées **au jour d'émission J** ; le recalcul de J après clôture (via `upsertDailyStats`) incorpore ces durées — donc le jour J n'est « définitif » qu'une fois tous ses tickets terminés ou expirés (fenêtre de rétro-calcul documentée — voir `_notes.md`).
- **INDÉSIRABLE (tickets en cours)** — SI un ticket est encore ouvert (`WAITING`/`CALLED`/`SERVING`) au moment de l'agrégation, ALORS il n'entre dans AUCUN numérateur de durée (pas de `closed_at`), mais compte dans `tickets_issued` ; l'occupation en cours n'est pas figée.
- **INDÉSIRABLE (scope network)** — SI `scope=network`, ALORS la réponse ne contient AUCUNE donnée personnelle (schéma `AnonymizedNetworkAggregate` de CONTRACT-006) — agrégats purs uniquement.
