## Exigences (EARS)
- `POST /tickets/sync` † (session borne) : `TicketSyncRequest` = `{ tickets: TicketSyncItem[] }`, batch ≤100 (`BATCH_TOO_LARGE` 422 sinon). **LA LOI (core.yaml)** — `TicketSyncItem` (`additionalProperties:false`) = `{ localUuid (uuid), serviceId (uuid), channel (TicketChannel), createdOfflineAt (date-time), priority? (TicketPriority) }`. Pas de `téléphone`/`localNumber` dans la sync. QUAND un localUuid est déjà connu, la ligne est `skipped {reason: ALREADY_SYNCED}` — rejouer le batch entier = même réponse, zéro doublon (contrainte unique DB + upsert). `X-Idempotency-Key` obligatoire (400 `IDEMPOTENCY_KEY_REQUIRED` ; 409 `IDEMPOTENCY_CONFLICT` si même clé/payload différent).
- Résolution des numéros : les tickets offline reçoivent des numéros DÉFINITIFS serveur (lock-then-increment) dans l'ordre `createdOfflineAt` ; **`TicketSyncResponse` = `{ synced: SyncedTicket[], skipped: SkippedTicket[] }`** avec `SyncedTicket = {localUuid, serverId, number}` (LA LOI — PAS de displayNumber/trackingId ici) et `SkippedTicket = {localUuid, reason}` (`reason` = string libre). La borne réaffiche `number`.
- SI `createdOfflineAt` est dans le futur (>5 min de dérive) → `skipped {reason: CLOCK_SKEW}` (jamais de rejet du batch entier) ; SI le service n'existe plus → `skipped {reason: SERVICE_NOT_FOUND}` (code aligné sur l'exemple du contrat, pas « UNKNOWN_SERVICE »).
- Les tickets synchronisés entrent en file WAITING (positions recalculées) et émettent `queue:updated` UNE fois par file affectée (pas par ticket).
- Transaction par TICKET (pas par batch) : un échec isolé → skipped, le reste passe ; tout est journalisé Pino.
- **QUAND des lignes sont skipped à la sync**, le SERVEUR émet **`alert:manager` de type `KIOSK_SYSTEM_ERROR`** (une alerte par batch, payload = nombre de skipped + raisons agrégées) — c'est le serveur qui émet cette alerte, pas la borne (voir CONTRACT-012).
- Suite `offline-resilience` (F0) branchée sur l'API réelle : rejeu double, crash mi-batch simulé (coupure) → reprise idempotente.
