# API-005 : Sync offline — batch borne idempotent

**Module** : F3 — API · **Agent** : agent-api · **Dépend de** : API-003 · **Statut** : TODO

**Révision** : v2 — arbitrage 19

## Exigences (EARS)
- `POST /tickets/sync` † (session borne) : batch ≤100 (`BATCH_TOO_LARGE` sinon) de tickets locaux `{localUuid, service, priorité, téléphone?, issuedAtLocal, localNumber}` ; QUAND un localUuid est déjà connu, la ligne est `skipped {reason: ALREADY_SYNCED}` — rejouer le batch entier = même réponse, zéro doublon (contrainte unique DB + upsert).
- Résolution des numéros : les tickets offline reçoivent des numéros DÉFINITIFS serveur (lock-then-increment) dans l'ordre `issuedAtLocal` ; la réponse mappe `{localUuid, serverId, number, displayNumber, trackingId}` — la borne réaffiche.
- SI `issuedAtLocal` est dans le futur (>5 min de dérive) → `skipped {reason: CLOCK_SKEW}` (jamais de rejet du batch entier) ; SI le service n'existe plus → `skipped {reason: UNKNOWN_SERVICE}`.
- Les tickets synchronisés entrent en file WAITING (positions recalculées) et émettent `queue:updated` UNE fois par file affectée (pas par ticket).
- Transaction par TICKET (pas par batch) : un échec isolé → skipped, le reste passe ; tout est journalisé Pino.
- **QUAND des lignes sont skipped à la sync**, le SERVEUR émet **`alert:manager` de type `KIOSK_SYSTEM_ERROR`** (une alerte par batch, payload = nombre de skipped + raisons agrégées) — c'est le serveur qui émet cette alerte, pas la borne (voir CONTRACT-012).
- Suite `offline-resilience` (F0) branchée sur l'API réelle : rejeu double, crash mi-batch simulé (coupure) → reprise idempotente.
