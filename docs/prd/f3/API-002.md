# API-002 : Middleware tenant + RBAC — le JWT fait loi

**Révision** : v2 — arbitrage 19

## Exigences (EARS)
- UBIQUITAIRE : toute route non-publique passe par `tenantMiddleware` — vérifie le JWT, ouvre le contexte via `withTenant(bankId)` (connexion sigfa_app) et expose `c.var.tenant` typé ; AUCUN handler ne reçoit de connexion hors contexte (revue + test structurel : grep des accès DB directs).
- Le middleware doit enforcer **x-required-role** : la matrice vient de `rbac-matrix.ts` (DB-003) + un mapping route→rôle GÉNÉRÉ depuis les bundles YAML (test d'alignement : chaque route du contrat a son entrée, toute route sans mapping → 500 au démarrage, jamais de défaut permissif).
- SI le rôle est insuffisant → 403 ; SI le token est expiré/invalide → 401 ; SI la route est `platform` (SUPER_ADMIN), ALORS connexion **`withPlatform`** de `@sigfa/database` (connexion migrateur dédiée, bankId non requis, couture périmètre étendu consignée) — routes listées exhaustivement. **Jamais de SET bank_id vide.**
- LÀ OÙ la route exige `AUTHENTICATED`, tout JWT valide passe (dont session borne scope agency).
- SI un payload contient bankId/agencyId incohérent avec le JWT (routes x-security-note : /queues…), ALORS 403 — helpers `assertAgencyInScope`.
- Chaque requête est journalisée Pino avec requestId + bankId (jamais de token ni téléphone en clair dans les logs).
