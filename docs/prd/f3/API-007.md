# API-007 : Agents — statuts temps réel, chronomètre, alertes manager

**Révision** : v2 — arbitrage 19

## Exigences (EARS)
- `POST /agents/:id/status` : machine à états AVAILABLE/SERVING/PAUSED/ABSENT/OFFLINE (LA LOI agents.yaml, 409 dédié sur transition illégale — dont SERVING→ABSENT avec ticket ouvert sans transfert) ; chaque changement écrit `agent_status_history` + émet `counter:status`.
- SERVING est PILOTÉ par le cycle ticket (call→SERVING à serve, →AVAILABLE à close/no-show) — le poste ne force pas SERVING manuellement (409).
- `GET /agents/:id/stats` : tickets traités jour, TMT moyen jour, ticket en cours (numéro + durée chronométrée) — règle « self » : un AGENT ne lit que les siennes, MANAGER+ celles de son scope.
- **Anti-flap par agentId** : à la déconnexion socket, Redis `SET NX TTL 30 s` par `agentId` — reconnexion dans la fenêtre → `DEL` (aucun effet). Au-delà : traitement AGENT_DISCONNECTED_WITH_TICKET.
- **Alertes** (`alert:manager`, types LA LOI, room de l'agence, + persistées ? NON — instantanées, l'agrégat = F10 ; documenté) :
  - AGENT_INACTIVE : AVAILABLE sans appel depuis `agent_inactivity_minutes` (banks) — vérification par **job BullMQ repeatable** `inactive-agent-scan` (verrou distribué, `worker concurrency=1`) ; intervalle exporté depuis `config/alerting.ts`, injectable via env. UNE alerte par épisode.
  - SLA_BREACH : ticket SERVING dépassant le SLA du service — vérification par **job BullMQ repeatable** `sla-scan` (même verrou distribué) ; >2× SLA → alerte renouvelée marquée `escalated`.
  - AGENT_DISCONNECTED_WITH_TICKET : QUAND le socket d'un agent avec ticket CALLED/SERVING se déconnecte >30 s (grâce anti-flap par agentId), ALORS le ticket repasse **WAITING priorité PRIORITY** (v5), `counter:status` OFFLINE, alerte immédiate.
- Déconnexion socket SANS ticket → statut OFFLINE simple après la même grâce anti-flap.
