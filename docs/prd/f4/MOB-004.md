## MOB-004 : Push "plus que 2 personnes" + offline MMKV + sync queue

**Module** : F4 · **Agent** : agent-mobile · **Dépend de** : MOB-003 · **Statut** : TODO

**Révision** : v2 — arbitrage 19

---

### Exigences (EARS)

- QUAND l'utilisateur installe l'app, le système doit enregistrer le device push via `POST /notifications/devices` (plateforme `EXPO` ou `IOS`/`ANDROID` selon le runtime) — idempotent : même token → même `deviceId`, aucun doublon.
- QUAND le device est supprimé ou l'utilisateur se désinscrit, le système doit appeler `DELETE /notifications/devices/{deviceId}`.
- QUAND la position dans la file passe à ≤ 2, le système doit émettre une notification push via Expo Notifications + FCM avec le message : *« Plus que 2 personnes devant vous — dirigez-vous vers l'agence »* + temps de trajet estimé si géoloc accordée.
- Le système doit persister en MMKV : `trackingId`, `position`, `estimatedWaitMinutes`, `lastSyncAt` — la TTL est la durée de vie du ticket (clôture = purge MMKV).
- QUAND la connexion réseau est perdue après l'émission du ticket, le système doit conserver la dernière position connue en MMKV et afficher un badge offline discret (`--info` #2E90FA). MOB-004 implémente **`flush()`** : consommation FIFO de `pending_tickets[]` MMKV (déduplication par `X-Idempotency-Key` + purge des tickets clôturés) au retour réseau.
- SI l'opt-in push n'est pas accordé par l'utilisateur, ALORS le parcours doit fonctionner en mode polling seul (aucune fonctionnalité bloquée).
- Le système doit gérer 5 états par écran : nominal, loading (enregistrement device), empty (aucun ticket actif), error (échec enregistrement → retry silencieux), offline (MMKV).
