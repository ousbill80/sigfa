### Exigences (EARS)

**UBIQUITAIRE** : L'application Next.js 15 App Router doit injecter en tête de chaque layout les variables CSS du Design System (`--surface-0`, `--surface-1`, `--ink-strong`, `--ink-soft`, `--success`, `--warning`, `--danger`, `--info`, `--brand`, `--brand-soft`, `--brand-contrast`) lues depuis le profil tenant.

**UBIQUITAIRE** : Les 6 rôles (`SUPER_ADMIN`, `BANK_ADMIN`, `AGENCY_DIRECTOR`, `MANAGER`, `AGENT`, `AUDITOR`) sont stockés dans le claim JWT et vérifiés par un middleware de layout côté serveur (RSC). Aucun composant client ne prend de décision d'accès.

**QUAND** un utilisateur accède à une route protégée sans JWT valide, le système doit le rediriger vers `/login` avec le paramètre `?next=` pour restauration post-login.

**QUAND** un JWT expire, le shell doit rafraîchir le token silencieusement via le refresh token (rotation) avant l'expiration des 15 minutes ; si le refresh échoue, rediriger vers `/login`.

**QUAND** un utilisateur authentifié tente d'accéder à une route interdite pour son rôle (matrice MODULE 4), le système doit retourner une page 403 avec lien vers son dashboard autorisé.

**SI** le profil tenant change (webhook ou reconnexion), le shell doit mettre à jour les variables CSS `--brand*` sans rechargement de page.

**Anormal** : SI l'API mock Prism ne répond pas au healthcheck initial, le shell doit afficher une page d'erreur "Service indisponible" et loguer l'URL cible.
