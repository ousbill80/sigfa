## MOB-001 : Shell Expo — navigation, i18n, tokens, auth légère par téléphone

**Module** : F4 · **Agent** : agent-mobile · **Dépend de** : CONTRACT-009 · **Statut** : TODO

**Révision** : v2 — arbitrage 19

---

### Exigences (EARS)

- Le shell doit bootstrapper Expo SDK 51 + Expo Router v3 (routes typées, file-based) avec une structure `app/` conforme au gabarit F4.
- Le shell doit appliquer les tokens SIGFA (`--brand`, `--surface-0`, `--ink-strong`, rayons 12px, grille 4px) via un fichier `tokens.ts` partagé — aucune valeur en dur dans les composants.
- Le shell doit exposer i18n 4 langues (FR / Dioula / Baoulé / EN) via `expo-localization` + `i18n-js` ; aucun texte en dur ; les 4 langues rendues sans débordement sur cibles ≥ 44×44px.
- QUAND un utilisateur se connecte, le shell doit proposer une auth légère par numéro de téléphone (OTP SMS) avec opt-in UEMOA explicite — l'UUID interne n'apparaît jamais ; seul le `trackingId` nanoid(21) circule côté mobile.
- Le shell doit gérer 5 états par écran : nominal, loading, empty, error, offline — le badge offline est discret (jamais bloquant).
- CONTRAINTE DÉCISION Metro/pnpm (leçon F0) — **Plan B documenté, ordre de tentative** :
  1. `metro.config.js` `watchFolders` + `extraNodeModules` (hoisting sélectif, pas d'impact monorepo).
  2. Sinon : symlinks `postinstall` ciblés (documentés dans `apps/mobile/scripts/`).
  3. Sinon : `node-linker=hoisted` GLOBAL avec **gate orchestrateur** (jamais silencieux — décision consignée dans `apps/mobile/metro.config.js`).
  Le résultat final (option choisie + justification) est documenté dans `apps/mobile/metro.config.js`.
- Le shell doit coder contre le mock Prism (CONTRACT-009) via la variable d'env `EXPO_PUBLIC_API_URL` — aucun fetch vers une route absente du contrat.
