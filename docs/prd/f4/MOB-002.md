## MOB-002 : Prise de ticket à distance — parcours 3 étapes

**Module** : F4 · **Agent** : agent-mobile · **Dépend de** : MOB-001 · **Statut** : TODO

**Révision** : v2 — arbitrage 19

---

### Exigences (EARS)

- QUAND un utilisateur lance la prise de ticket, le parcours doit reproduire exactement les 3 étapes du kiosque : (1) choix de l'agence/service, (2) confirmation avec téléphone pré-rempli et opt-in UEMOA, (3) écran de confirmation du ticket.
- Le système doit appeler `POST /public/tickets` (canal `MOBILE`) avec `X-Idempotency-Key` nanoid(21) généré côté client — rejeu de la même clé dans les 24h retourne le ticket original sans doublon.
- QUAND l'émission réussit, l'écran de confirmation doit afficher : `displayNumber` (format `{code}-{NNN}`), `position`, `estimatedWaitMinutes` — en tokens typographiques Display (≥ 32px sur mobile).
- QUAND `estimatedWaitMinutes` est disponible, le parcours doit afficher l'attente estimée par service en temps réel (polling `GET /public/tickets/{trackingId}` cache 30s, `ETag` supporté).
- SI la connexion réseau est absente au moment de l'émission, ALORS le système doit appeler **`enqueue()`** pour persister la requête dans `pending_tickets[]` MMKV et l'utilisateur voit un écran de confirmation optimiste avec badge "Synchronisation en cours". Le **`flush()`** FIFO (dédupliqué + purge) est implémenté dans MOB-004.
- Le système doit respecter les contraintes de sécurité du contrat : numéro de téléphone jamais exposé en clair dans les logs ; seul `trackingId` stocké localement en MMKV.
