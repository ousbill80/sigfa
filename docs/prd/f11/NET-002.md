## NET-002 : Mise à jour centralisée des bornes (1 déploiement → toutes agences) + canary

**Agent** : direct · **Module** : F11 · **Dépend de** : ADM-003 · **Statut** : TODO

**Révision** : v1 — expansion Boucle 1 (F11).

> Story d'ops/déploiement (pas de PII, pas de cross-tenant lecture). S'appuie sur le heartbeat borne existant (`app_version`, `last_seen`, `printer_status` — API-011) et la supervision `GET /kiosks/status` (ADM-003). L'objet du déploiement = l'application borne (Electron/kiosk), pas les données métier.

---

### Exigences (EARS)

**UBIQUITAIRE** : Le système doit permettre de publier UNE version de l'application borne (artefact signé) et de la déployer vers le parc de bornes, sans intervention physique par borne. Une borne connaît sa version courante via le heartbeat (`app_version` — API-011).

**UBIQUITAIRE — preuve d'intégrité borne** : chaque artefact de version doit porter une signature/hash (checksum) vérifiée par la borne AVANT application. SI le hash de l'artefact téléchargé ne correspond pas à la signature attendue, ALORS la borne refuse la mise à jour, reste sur la version courante, et remonte un événement d'échec d'intégrité (visible en supervision). Aucune version non signée n'est appliquée.

**QUAND** une nouvelle version est publiée, le système doit exécuter un **rollout canary** en cohortes croissantes explicites : cohorte 1 = **canary ≤5% du parc** (bornes désignées), puis paliers (ex. 25% → 50% → 100%) déclenchés MANUELLEMENT ou après fenêtre de stabilité verte. La stratégie de cohorte est déterministe et traçable (quelle borne, quel palier, quand).

**QUAND** le canary est en cours, le système doit surveiller la santé des bornes de la cohorte via le heartbeat + supervision (ADM-003) : version effective adoptée, taux de bornes muettes (`OFFLINE`), taux d'erreur imprimante (`printer_status`), crash/redémarrage. Un palier ne progresse QUE si la cohorte reste sous les seuils de santé.

**SI** une cohorte canary dépasse un seuil d'échec (défaut : **>10% des bornes de la cohorte OFFLINE ou en échec d'intégrité/crash dans la fenêtre de stabilité**), ALORS le rollout doit se **STOPPER automatiquement** (halt), NE PAS progresser au palier suivant, et déclencher une alerte (destinataires : ops plateforme — cohérent NET-003).

**SI** un rollback est demandé (manuel) ou déclenché (halt), ALORS le système doit **repointer les cohortes affectées vers la version stable précédente** (rollback), les bornes redescendent à la version antérieure connue-bonne, et la supervision confirme le retour (heartbeat `app_version` = version stable). La version stable précédente est toujours conservée et redéployable.

**LÀ OÙ** une borne est OFFLINE au moment d'un palier, elle doit adopter la version cible de son palier au prochain heartbeat/reconnexion (idempotence : réappliquer la même version cible est sans effet), jamais sauter de version de façon incohérente.

**anormal** : une borne dont le hash d'intégrité échoue de façon répétée (défaut : 3 tentatives) est marquée en quarantaine de mise à jour (reste sur la version stable) et signalée en supervision pour intervention — elle n'entre pas en boucle de téléchargement.

### Critères d'acceptation

- [ ] `NET-002: publication d'une version signée — artefact non signé/hash invalide → refus d'application, borne reste sur version courante (test intégrité)`
- [ ] `NET-002: rollout canary en cohortes — canary ≤5% d'abord, paliers 25/50/100% déterministes et tracés (test séquencement cohortes)`
- [ ] `NET-002: progression d'un palier UNIQUEMENT si cohorte sous seuils de santé (version adoptée, OFFLINE, crash) — sinon pas de progression`
- [ ] `NET-002: cohorte > seuil d'échec (>10% OFFLINE/échec/crash) → halt automatique + alerte ops (test)`
- [ ] `NET-002: rollback → cohortes repointées sur version stable précédente, heartbeat app_version = version stable confirmé en supervision`
- [ ] `NET-002: idempotence — réappliquer la même version cible à une borne est sans effet ; borne OFFLINE adopte la cible à la reconnexion`
- [ ] `NET-002: quarantaine — 3 échecs d'intégrité → borne reste sur version stable + signalée, pas de boucle de téléchargement`
- [ ] `NET-002: aucune donnée métier/PII manipulée — le déploiement porte l'artefact applicatif borne uniquement (revue)`
