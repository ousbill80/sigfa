## NET-003 : Monitoring production — Grafana, Sentry, alertes CPU>80% / mem>85% / err>1%

**Agent** : direct · **Module** : F11 · **Dépend de** : SEC-004 · **Statut** : TODO

**Révision** : v1 — expansion Boucle 1 (F11).

> Story d'observabilité infra (pas de cross-tenant lecture, pas de PII exposée dans les métriques). S'appuie sur la campagne de charge k6 (SEC-004 : Socket.io p95 <500ms, 100 tickets/min/agence × 50 agences) pour calibrer les seuils et valider les dashboards sous charge.

---

### Exigences (EARS)

**UBIQUITAIRE** : La production doit exposer des métriques (API, Socket.io, PostgreSQL, Redis/BullMQ, bornes) collectées dans **Grafana** (dashboards) et les erreurs applicatives capturées dans **Sentry**. Les métriques sont techniques (latence, débit, CPU, mémoire, taux d'erreur, profondeur de file BullMQ) — **jamais** de PII client ni de contenu métier dans les labels/traces (les téléphones/tracking_id sont exclus des breadcrumbs Sentry — scrubbing obligatoire).

**UBIQUITAIRE — dashboards Grafana** : au minimum, un dashboard par domaine :
- **API** : latence p50/p95/p99 par route, taux d'erreur (5xx), débit ;
- **Temps réel** : latence de livraison `ticket:called` (cohérent SEC-004 p95 <500ms), connexions Socket.io actives ;
- **Infra** : CPU, mémoire, PgBouncer/pool DB, Redis, profondeur/retry/dead-letter BullMQ ;
- **Parc bornes** : nombre ONLINE/OFFLINE (dérivé ADM-003), échecs de rollout (cohérent NET-002).

**QUAND** l'utilisation CPU d'un service dépasse **80%** sur une fenêtre soutenue (défaut : 5 min), le système doit déclencher une alerte `WARNING` vers les destinataires ops.

**QUAND** l'utilisation mémoire d'un service dépasse **85%** sur la fenêtre (défaut : 5 min), le système doit déclencher une alerte `WARNING` vers les destinataires ops.

**QUAND** le taux d'erreur applicatif (5xx / requêtes) dépasse **1%** sur la fenêtre (défaut : 5 min), le système doit déclencher une alerte `CRITICAL` vers les destinataires ops (astreinte).

**QUAND** la latence de livraison temps réel `ticket:called` dépasse le SLA (**p95 ≥ 500ms** — cohérent SEC-004), le système doit déclencher une alerte de dégradation (le seuil produit est une contrainte de sortie).

**SI** une dépendance critique tombe (PostgreSQL, Redis) ou que `GET /health` renvoie 503 (`SERVICE_UNAVAILABLE` — API-011), ALORS une alerte `CRITICAL` immédiate part vers l'astreinte (pas d'attente de fenêtre).

**LÀ OÙ** une alerte est levée, elle doit être routée par sévérité : `WARNING` → canal ops (email/Slack ops), `CRITICAL` → astreinte on-call (email + push/téléphone). Chaque règle d'alerte nomme explicitement son/ses destinataire(s) et sa fenêtre.

**anormal** : le système d'alerte doit éviter les tempêtes d'alertes (dédup / regroupement / seuil de flapping) — une même condition soutenue ne génère pas une alerte par échantillon.
