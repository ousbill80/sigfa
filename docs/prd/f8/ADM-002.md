# ADM-002 : Onboarding nouvelle agence < 2h — template, clonage, accès borne + QR d'installation

**Module** : F8 — ADMIN & THEMING · **Dépend de** : API-009, WEB-006 · **Statut** : TODO

**Révision** : v1 — expansion Boucle 1 (Fable 5)

> **Objectif produit mesurable** (critère de sortie PRD §4) : **onboarding agence < 2 heures**. Ici « onboarding » = de la création de l'agence à une borne opérationnelle imprimant un premier ticket. On rend le parcours **chronométrable** : chaque étape a un temps cible ; la somme des temps cibles est < 2h et le parcours mémorise `startedAt`/`completedAt`.

> **Découpage (Loi 4)** : ADM-002a (agent-api — clonage + génération accès borne/QR) et ADM-002b (agent-web — parcours guidé chronométré). Front sur mock Prism ; réel = RT-001.

## ADM-002a — API : template, clonage config, accès borne, QR d'installation (agent-api)

**Agent** : agent-api · **Dépend de** : API-009

### Exigences (EARS)

**UBIQUITAIRE** : L'API doit exposer sous `admin.yaml` :
- `GET /api/v1/banks/{bankId}/agency-templates` (BANK_ADMIN+) → templates d'agence disponibles (par défaut : template « agence standard » livré).
- `POST /api/v1/banks/{bankId}/agencies:clone` (BANK_ADMIN+) → corps `{ name, templateId | sourceAgencyId, overrides? }` → crée une agence en **clonant** services, SLA, priorités, guichets, seuils depuis le template ou une agence source ; retourne l'agence créée + un `onboardingId`.
- `POST /api/v1/agencies/{agencyId}/kiosks:provision` (AGENCY_DIRECTOR+) → génère un **accès borne** : `{ kioskId, enrollmentToken, enrollmentQrUrl, expiresAt }`.
- `GET /api/v1/agencies/{agencyId}/onboarding/{onboardingId}` → état du parcours + timestamps.

**UBIQUITAIRE** : Le clonage doit être **structurel uniquement** : il copie la configuration métier (services/SLA/guichets/seuils), **jamais** de données clients, tickets, ni PII de l'agence source. Le `--brand`/theme de la banque est hérité du tenant (ADM-001), pas cloné par agence.

**QUAND** un clonage réussit, le système doit horodater `onboarding.startedAt` et matérialiser les **étapes** avec statut (`agency_created`, `services_cloned`, `counters_ready`, `agents_imported`, `kiosk_provisioned`) — chacune complétable et horodatée, pour rendre le < 2h mesurable.

**QUAND** un accès borne est provisionné, le système doit générer un **`enrollmentToken` à usage unique, à courte durée de vie (défaut 60 min, configurable), lié au `kioskId` et au tenant**, et un QR encodant l'URL d'enrôlement (jamais le token en clair dans un log). Le durcissement du token borne suit le contrat public/build déjà clos (cf. mémoire — token TV/borne durci).

**QUAND** une borne s'enrôle avec un `enrollmentToken` valide et non expiré, le système échange le token contre les credentials borne persistants et **invalide** le token d'enrôlement (usage unique).

**SI** un `enrollmentToken` est expiré, déjà consommé, ou d'un autre tenant → 401/403 opaque `KIOSK_ENROLLMENT_INVALID` ; jamais de détail distinguant expiré vs inconnu.

**Anormal** : `:clone` avec `templateId` ET `sourceAgencyId` absents des deux → 422 `CLONE_SOURCE_REQUIRED`. `sourceAgencyId` hors tenant → 404 opaque.

### Critères d'acceptation

- [ ] `ADM-002a: clone crée agence avec services/SLA/guichets/seuils du template, zéro ticket/PII copié`
- [ ] `ADM-002a: clone hérite theme tenant (ADM-001), ne clone pas de brand par agence`
- [ ] `ADM-002a: onboarding matérialise 5 étapes horodatées (startedAt/completedAt par étape)`
- [ ] `ADM-002a: kiosks:provision → enrollmentToken usage-unique + QR + expiresAt (défaut 60min)`
- [ ] `ADM-002a: enrôlement borne consomme et invalide le token (rejeu → KIOSK_ENROLLMENT_INVALID)`
- [ ] `ADM-002a: token expiré/inconnu/autre tenant → 401/403 opaque, indistinct`
- [ ] `ADM-002a: enrollmentToken absent des logs (test scrub de log)`
- [ ] `ADM-002a: clone d'une agence d'un autre tenant → 404 opaque (tenant-isolation)`
- [ ] `ADM-002a: clone sans templateId ni sourceAgencyId → 422 CLONE_SOURCE_REQUIRED`
- [ ] `ADM-002a: X-Idempotency-Key sur :clone et :provision (rejeu → même réponse)`
- [ ] `ADM-002a: Schemathesis PASS sur routes templates/clone/provision/onboarding`

## ADM-002b — Web : parcours guidé chronométré + QR d'installation (agent-web)

**Agent** : agent-web · **Dépend de** : WEB-006, ADM-002a

### Exigences (EARS)

**UBIQUITAIRE** : La console admin doit étendre l'onboarding de WEB-006 en un **parcours guidé Stepper** (Design System v2 §4 — composant Stepper) de 5 étapes : (1) Créer/nommer l'agence à partir d'un template, (2) Vérifier/ajuster services & SLA clonés, (3) Vérifier guichets, (4) Importer agents (CSV, réutilise WEB-006), (5) Provisionner la borne → afficher le **QR d'installation**.

**UBIQUITAIRE** : Chaque étape affiche son **temps cible** et le parcours affiche un **chronomètre global** (`startedAt` → maintenant) ainsi qu'un indicateur « < 2h » (vert tant que sous cible). Le parcours est reprenable : quitter puis revenir restaure l'étape courante depuis `GET .../onboarding/{id}`.

**QUAND** l'admin atteint l'étape 5, la console affiche le **QR d'installation** (grand, imprimable) + instructions FR/EN pour scanner depuis la borne, et l'expiration du token. Un bouton « Régénérer » réémet un token si expiré.

**QUAND** toutes les étapes sont complètes, le parcours affiche un récapitulatif avec la **durée totale mesurée** et un état « Agence opérationnelle ».

**SI** une étape échoue (validation, API), le Stepper bloque l'avancement, affiche l'erreur inline, et conserve la progression des étapes déjà validées.

**LÀ OÙ** le rôle n'est pas BANK_ADMIN+ (étapes 1-2) ou AGENCY_DIRECTOR+ (étapes 3-5 sur l'agence), les actions correspondantes sont désactivées/403 (RBAC MODULE 4).

**Anormal** : SI le QR ne peut être généré, message « Borne non provisionnée : {motif} » + bouton réessayer ; jamais d'étape 5 apparemment terminée sans QR valide.

### Critères d'acceptation

- [ ] `ADM-002b: Stepper 5 étapes — parcours complet testable (Testing Library)`
- [ ] `ADM-002b: chaque étape affiche temps cible ; chronomètre global + indicateur < 2h`
- [ ] `ADM-002b: parcours reprenable — quitter/revenir restaure l'étape courante depuis onboarding/{id}`
- [ ] `ADM-002b: étape 5 affiche QR imprimable + expiration + régénérer`
- [ ] `ADM-002b: récap final affiche durée totale mesurée + « Agence opérationnelle »`
- [ ] `ADM-002b: étape en échec → avancement bloqué, erreur inline, progression conservée`
- [ ] `ADM-002b: RBAC — actions étapes désactivées/403 selon rôle`
- [ ] `ADM-002b: QR non généré → message + réessayer (jamais d'étape 5 sans QR)`
- [ ] `ADM-002b: états loading/error/offline (offline → « Connexion requise pour l'onboarding »)`
- [ ] `ADM-002b: tokens uniquement — axe-core PASS`
