# ADM-001 : Theming par tenant — logo, `--brand` contraste auto ≥4.5:1, messages d'accueil

**Module** : F8 — ADMIN & THEMING · **Dépend de** : WEB-001, API-008 · **Statut** : TODO

**Révision** : v1 — expansion Boucle 1 (Fable 5)

> **Principe cardinal (Loi 5 v1 + Design System v2 §1 + mémoire « theming banque facile »)** : le theming est **HABILLAGE, JAMAIS STRUCTURE**. Un tenant ne change QUE : (a) `--brand` (**1 seul token**, dérivés calculés) ; (b) le logo ; (c) les messages d'accueil (FR/EN). Layout, grille, composants, `--forest`/`--gold`/sémantiques, ergonomie = **identiques pour toutes les banques**. Aucune option de mise en page par tenant n'existe et n'existera jamais.

> **Découpage (Loi 4 — une story = une couche = un agent)** : ADM-001 est scindée en **ADM-001a (agent-api)** et **ADM-001b (agent-web)**. Les deux consomment le contrat `admin.yaml` étendu (voir `_notes.md` → ajouts de contrat). Le front code contre le mock Prism ; sockets/backend réels = RT-001.

## ADM-001a — API : persistance & service de theming tenant (agent-api)

**Agent** : agent-api · **Dépend de** : API-008

### Exigences (EARS)

**UBIQUITAIRE** : L'API doit exposer la ressource `theme` d'un tenant sous `admin.yaml` :
- `GET /api/v1/banks/{bankId}/theme` → `{ brand, logoUrl?, welcomeMessages: { fr, en } }` (lecture BANK_ADMIN+, ET lecture publique projetée — voir plus bas).
- `PATCH /api/v1/banks/{bankId}/theme` (BANK_ADMIN+) → met à jour `brand` (hex `^#[0-9A-Fa-f]{6}$`), `welcomeMessages`, `logoUrl`.
- `POST /api/v1/banks/{bankId}/theme/logo` (BANK_ADMIN+) → upload logo (stockage objet R2), retourne `logoUrl`.

**UBIQUITAIRE** : Le token de theming tenant est **`brand` UNIQUEMENT**. L'API ne persiste ni n'accepte de token de structure, de layout, de police, d'espacement ni de rayon par tenant. Toute propriété inconnue dans le corps `PATCH` → 422 `UNKNOWN_FIELD` (le contrat n'expose PAS ces champs).

**QUAND** un `PATCH .../theme` fournit un `brand`, le système doit calculer et persister **les dérivés serveur-autoritaires** : `brandStrong` (variante pressé), `brandSoft` (fond de badge), et `brandContrast` — cette dernière garantissant un ratio de contraste WCAG **≥ 4,5:1** de `brandContrast` sur `brand` ET du texte sur les surfaces. Le calcul est déterministe et centralisé (utilitaire contraste de `@sigfa/ui`, réutilisé côté serveur).

**QUAND** le `brand` saisi ne permet aucun `brandContrast` ∈ {`--ink`, `--ink-inverse`} atteignant 4,5:1 sur `brand`, le système doit **assombrir automatiquement** `brand` (vers `brandStrong`) par pas déterministes jusqu'à ce qu'un contraste conforme existe, persister la valeur **corrigée**, et retourner dans la réponse `{ brandRequested, brand (corrigé), corrected: true }`.

**QUAND** un logo est uploadé, le système doit valider type (`image/png`, `image/svg+xml`, `image/webp`), taille (≤ 512 Ko), et dimensions ; SI invalide → 422 `INVALID_LOGO`. Aucune exécution de contenu SVG (sanitisation).

**QUAND** le theme d'un tenant change, le système doit invalider le cache de la projection publique du theme (consommée par kiosque/web/TV) pour ce `bankId` uniquement (isolation tenant).

**LÀ OÙ** l'appelant n'est pas authentifié (role NONE), l'API doit exposer une projection publique en lecture seule `GET /public/banks/{bankId}/theme` → **UNIQUEMENT** `{ brand, brandStrong, brandSoft, brandContrast, logoUrl?, welcomeMessages }` (zéro PII, zéro config interne).

**Anormal** : `PATCH .../theme` avec `brand` non hex → 422 `INVALID_BRAND` opaque et déterministe.

### Critères d'acceptation

- [ ] `ADM-001a: PATCH theme persiste brand + dérivés brandStrong/brandSoft/brandContrast calculés serveur`
- [ ] `ADM-001a: brandContrast garantit ratio ≥ 4.5:1 sur brand (test contraste sur 20 couleurs échantillon)`
- [ ] `ADM-001a: brand non conforme → brand assombri automatiquement, réponse corrected:true + brandRequested`
- [ ] `ADM-001a: corps PATCH avec champ de structure/layout → 422 UNKNOWN_FIELD (aucun token hors brand accepté)`
- [ ] `ADM-001a: brand non hex → 422 INVALID_BRAND`
- [ ] `ADM-001a: upload logo type/taille invalide → 422 INVALID_LOGO ; SVG sanitisé (zéro script)`
- [ ] `ADM-001a: GET /public/banks/{id}/theme expose UNIQUEMENT brand*/logoUrl/welcomeMessages (zéro PII)`
- [ ] `ADM-001a: PATCH tenant A n'affecte pas le theme du tenant B (tenant-isolation)`
- [ ] `ADM-001a: changement theme invalide le cache projection publique du bankId seul`
- [ ] `ADM-001a: Schemathesis PASS sur les routes theme`

### Hors scope (ADM-001a)
- Rendu/injection CSS → ADM-001b.
- Onboarding / clonage template → ADM-002 (API-009).
- Supervision bornes → ADM-003 (API-011).

## ADM-001b — Web : injection CSS `--brand`, logo, messages d'accueil + éditeur (agent-web)

**Agent** : agent-web · **Dépend de** : WEB-001, ADM-001a (contrat `admin.yaml`)

### Exigences (EARS)

**UBIQUITAIRE** : Le shell (WEB-001) doit injecter au layout root les variables `--brand`, `--brand-strong`, `--brand-soft`, `--brand-contrast` **lues depuis la projection theme du tenant**. `--forest`, `--gold`, sémantiques et tous les autres tokens restent ceux du Design System v2 — **jamais surchargés par tenant**.

**UBIQUITAIRE** : La console admin (section « Identité banque », BANK_ADMIN+) doit fournir un éditeur de theming : sélecteur de couleur `--brand`, upload logo, éditeur de messages d'accueil FR/EN. Aucune commande de layout/structure n'est offerte à l'utilisateur.

**QUAND** le BANK_ADMIN saisit une couleur `--brand`, l'éditeur doit afficher **en temps réel une preview** du rendu (bouton primaire, badge, en-tête) ET le ratio de contraste calculé ; SI la couleur ne passe pas 4,5:1, un avertissement inline indique **la valeur corrigée qui sera appliquée** (miroir exact du calcul serveur ADM-001a — même utilitaire `@sigfa/ui`).

**QUAND** le BANK_ADMIN sauvegarde, la console envoie `PATCH .../theme` avec `X-Idempotency-Key` (UUID v4) et affiche la valeur **effectivement persistée** (potentiellement corrigée) sans rechargement de page.

**SI** le profil theme du tenant change (reconnexion / après save), le shell met à jour les variables `--brand*` **sans rechargement** (cohérent WEB-001).

**LÀ OÙ** le rôle est `AGENT`, `MANAGER` ou `AUDITOR`, la section « Identité banque » retourne 403 (RBAC matrice MODULE 4 — theming = BANK_ADMIN+).

**LÀ OÙ** aucun logo n'est défini, le système affiche un placeholder neutre (nom de la banque en typo Display) — jamais d'espace cassé.

**Anormal** : SI l'upload logo échoue (422/réseau), message inline « Logo non appliqué : {motif} », l'ancien logo reste actif.

### Critères d'acceptation

- [ ] `ADM-001b: --brand + dérivés injectés au layout root depuis projection tenant, snapshot 3 tenants`
- [ ] `ADM-001b: --forest/--gold/sémantiques JAMAIS surchargés par tenant (snapshot identique inter-tenants)`
- [ ] `ADM-001b: aucune commande de layout/structure exposée dans l'éditeur (test d'absence)`
- [ ] `ADM-001b: preview temps réel bouton/badge/en-tête au changement de --brand`
- [ ] `ADM-001b: --brand hors 4.5:1 → avertissement + valeur corrigée (identique au calcul serveur)`
- [ ] `ADM-001b: save → PATCH avec X-Idempotency-Key ; valeur persistée affichée sans rechargement`
- [ ] `ADM-001b: changement theme → --brand mis à jour sans rechargement (SocketProvider/reconnexion)`
- [ ] `ADM-001b: RBAC AGENT/MANAGER/AUDITOR → 403 sur section Identité banque`
- [ ] `ADM-001b: logo absent → placeholder nom banque (jamais layout cassé)`
- [ ] `ADM-001b: upload échoué → message inline, ancien logo conservé`
- [ ] `ADM-001b: états loading (skeleton) / error (inline) / offline (« Connexion requise pour configurer »)`
- [ ] `ADM-001b: tokens uniquement — axe-core PASS ; contraste ≥ 4.5:1 vérifié sur rendu 3 tenants`
