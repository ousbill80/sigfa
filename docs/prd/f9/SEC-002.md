# SEC-002 : Campagne tenant-isolation EXHAUSTIVE (table × route × attaque) + recâblage RLS armée

**Module** : F9 — Sécurité renforcée & charge · **Agent** : agent-database (périmètre : `packages/testing/src/tenant-isolation/` + `packages/database/src/rls/` ; couture API via dispatch coordonné agent-api) · **Dépend de** : F3 DONE, DB-002 (RLS `withTenant`/policies, DONE), RT-001 (réel branché) · **Statut** : TODO

**Révision** : v1 — expansion Boucle 1. **Résorbe la dette SEC-F3-02** consignée en clôture F3 : « recâbler l'accès DB via `withTenant`/RLS armée — l'isolation repose aujourd'hui sur le `WHERE bank_id` applicatif testé ». SEC-002 = campagne EXHAUSTIVE + bascule de l'isolation applicative vers l'isolation DÉFENSE-EN-PROFONDEUR (RLS forcée + `withTenant`), pas une simple extension de suite.

> **Ne duplique pas DB-002.** DB-002 a posé RLS `ENABLE`+`FORCE`+policy `tenant_isolation` sur chaque table à `bank_id`, le rôle applicatif non-owner non-BYPASSRLS, et l'utilitaire générique `assertTenantIsolated(table, fixtures)`. SEC-002 (a) **complète** la couverture à 100 % des tables des vagues F5–F8 ajoutées depuis, (b) **ferme SEC-F3-02** en garantissant que TOUT chemin d'accès DB de l'API passe par `withTenant`/RLS armée (plus aucune isolation reposant uniquement sur `WHERE bank_id` applicatif), (c) formalise une **matrice table × route × attaque** comme artefact vérifiable.

## Modèle de la matrice (artefact central)
`packages/testing/src/tenant-isolation/matrix.ts` : matrice générée **table × route × vecteur d'attaque**, source de vérité versionnée.
- **Axe TABLE** : 100 % des tables portant `bank_id` (introspection `information_schema` — le test échoue si une table à `bank_id` manque à la matrice).
- **Axe ROUTE** : 100 % des routes montées mutant ou lisant une table tenant (introspection du routeur Hono / OpenAPI bundle — une route non couverte → échec).
- **Axe ATTAQUE** (vecteurs, chacun un cas exécuté) :
  1. `CROSS_TENANT_READ` — contexte A, tenter de lire une ligne de B → zéro ligne.
  2. `CROSS_TENANT_WRITE` — contexte A, tenter UPDATE/DELETE d'une ligne de B → zéro effet.
  3. `INJECTED_BANK_ID_BODY` — contexte A, `bank_id` de B injecté dans le corps → policy `WITH CHECK` rejette.
  4. `INJECTED_BANK_ID_PARAM` — id de ressource de B en paramètre d'URL sous contexte A → 404 opaque (jamais 403 révélateur d'existence).
  5. `MISSING_CONTEXT` — aucun `app.current_bank_id` positionné → zéro ligne / écriture rejetée.
  6. `SQL_INJECTION` — payloads d'injection SQL sur champs filtrables/recherche (ex. `'; --`, `OR 1=1`, UNION) → paramétrage/échappement Drizzle, aucune fuite inter-tenant, aucune erreur 500 révélatrice.
  7. `PLATFORM_LEAK` — une route platform (`SUPER_ADMIN`, connexion migrateur) ne doit JAMAIS être atteignable sous un JWT de banque, et une route de banque ne doit jamais s'exécuter sur la connexion migrateur (BYPASSRLS).

## Exigences (EARS)

- **UBIQUITAIRE** : la suite `tenant-isolation` DOIT exécuter chaque cellule `(table, route, attaque)` applicable de la matrice sur PostgreSQL 16 réelle (Testcontainers), via la **connexion applicative `sigfa_app`** (non-owner, non-BYPASSRLS) — jamais l'owner (qui contourne FORCE RLS silencieusement, leçon DB-002).
- **Complétude par introspection** : la suite DOIT dériver la liste des tables de `information_schema` et la liste des routes du routeur monté ; **SI** une table à `bank_id` ou une route tenant n'a pas de couverture dans la matrice, ALORS le test ÉCHOUE (pas de couverture manuelle qui se périme).
- **Fermeture SEC-F3-02 (recâblage)** : TOUT accès DB de `apps/api` DOIT passer par `withTenant` (contexte SQL positionné) ou par la connexion platform explicitement listée ; un **test d'architecture** (grep/AST sur `apps/api/src`) interdit tout accès `db.*` métier hors `withTenant`/`withPlatform`. L'isolation ne repose PLUS sur un `WHERE bank_id` applicatif seul — elle est défense-en-profondeur (RLS forcée + contexte).
- **QUAND** un vecteur `CROSS_TENANT_*` est exécuté, le résultat DOIT être zéro ligne / zéro effet (jamais une erreur 500 qui divulguerait la structure) ; **QUAND** `INJECTED_BANK_ID_PARAM` cible une ressource d'un autre tenant, la réponse DOIT être 404 opaque (non-révélatrice d'existence).
- **anormal** : une injection SQL sur un champ filtrable ne doit produire NI fuite inter-tenant, NI 500, NI temps de réponse anormal révélateur — assertions explicites.
- **Régression permanente** : la matrice devient un gate de la CI (étape 6 de la gate de story « suite tenant-isolation si table/route touchée » élevée au niveau vague) — toute nouvelle table/route future rejoint la matrice ou casse la CI.

## Critères d'acceptation
- [ ] `SEC-002: introspection information_schema — 100% des tables à bank_id présentes dans la matrice (échec si manquante)`
- [ ] `SEC-002: introspection routeur — 100% des routes tenant couvertes par la matrice (échec si route non couverte)`
- [ ] `SEC-002: CROSS_TENANT_READ et CROSS_TENANT_WRITE → zéro ligne / zéro effet sur CHAQUE table (connexion sigfa_app, PG réelle)`
- [ ] `SEC-002: INJECTED_BANK_ID_BODY → rejet WITH CHECK ; INJECTED_BANK_ID_PARAM → 404 opaque (jamais 403 révélateur)`
- [ ] `SEC-002: MISSING_CONTEXT → zéro ligne / écriture rejetée sur chaque table`
- [ ] `SEC-002: SQL_INJECTION sur champs filtrables → aucune fuite inter-tenant, aucun 500, paramétrage vérifié`

- [ ] `SEC-002: test d'architecture — aucun accès db.* métier hors withTenant/withPlatform dans apps/api (SEC-F3-02 fermée)`
- [ ] `SEC-002: matrice branchée en gate CI — nouvelle table/route non couverte → CI rouge`
