# SEC-005 : Mutation testing Stryker ≥60% sur moteur de file + sla-engine

**Module** : F9 — Sécurité renforcée & charge · **Agent** : direct (qualité/tests) · **Dépend de** : REP-001 (F7 — moteur d'agrégats/suite `sla-engine`, **NON encore livré**), API-004 (DONE — moteur de file, suite `sla-engine` amorcée), API-003 (DONE — machine à états) · **Statut** : BLOCKED (attend REP-001)

**Révision** : v1 — expansion Boucle 1. Mesure la QUALITÉ des tests (pas la couverture de lignes) par mutation testing Stryker sur les deux cœurs métier critiques : le **moteur de file** (`selectNextTicket` : priorités VIP>PMR>SENIOR>PRIORITY>STANDARD, routage compétence+langue, débordement, pause — API-004) et le **sla-engine** (machine à états ticket + calculs TMA/TMT/TTS/abandon/SLA/NPS/occupation — API-003/004 + REP-001). Seuil : **score de mutation ≥ 60 %**.

> **Dépendance forward assumée.** `sla-engine` existe déjà comme suite (`packages/testing/src/sla-engine/`, étendue en API-004 : matrice priorités×langues×débordement, 100 % déterministe). Les CALCULS d'agrégats (TMA/TMT/TTS/NPS…) qui complètent le sla-engine arrivent avec **REP-001 (F7, non livré)**. SEC-005 est donc **BLOCKED tant que REP-001 n'est pas DONE** — sinon il mesurerait la mutation sur un moteur d'agrégats absent. L'orchestrateur ne dispatche SEC-005 qu'après REP-001 DONE (ou en scope réduit « moteur de file seul » sur décision explicite, cf. `_notes.md`).

## Cible chiffrée (verrouillée)
- **Score de mutation ≥ 60 %** = (mutants tués + timeout) / (mutants couverts, hors « no coverage » et hors mutants équivalents exclus explicitement), sur le périmètre `{ moteur de file, sla-engine }` UNIQUEMENT — pas sur tout le monorepo.
- Périmètre de mutation borné par config Stryker versionnée (`stryker.conf.*`) : `mutate` ciblé sur les fichiers du moteur de file et du sla-engine ; le reste du code est HORS champ de ce seuil.

## Exigences (EARS)

- **UBIQUITAIRE** : Stryker DOIT muter EXCLUSIVEMENT les fichiers du moteur de file (`selectNextTicket` & routage/débordement/pause, API-004) et du sla-engine (machine à états + calculs d'agrégats REP-001), et rapporter un **score de mutation ≥ 60 %** sur ce périmètre — le job échoue en dessous.
- **QUAND** Stryker s'exécute, ALORS il DOIT réutiliser les suites déterministes existantes (`sla-engine` table-driven, tests moteur de file) — pas de tests dédiés artificiels gonflant le score ; les mutants survivants pointent des tests réels manquants à AJOUTER, pas à masquer.
- **SI** un mutant est équivalent (ne change pas le comportement observable), ALORS il DOIT être exclu **explicitement et justifié** en config (liste versionnée) — l'exclusion est traçable, jamais un moyen d'atteindre 60 % artificiellement.
- **Déterminisme** : le run de mutation DOIT être reproductible (mêmes suites, mêmes seeds, timeouts calibrés) — un score flaky n'est pas recevable (cohérent avec la discipline anti-flake F3/RT-002).
- **anormal** : si le score chute sous 60 % après un changement du moteur/sla-engine, la CI DOIT échouer et lister les mutants survivants (rapport HTML/JSON archivé) pour guider l'ajout de tests.
- **Gate de sortie produit** : ce seuil matérialise « mutation ≥ 60 % cœur » des critères de sortie produit (PRD §4.6).

## Critères d'acceptation

## Hors scope de cette story
- Mutation testing hors périmètre `{moteur de file, sla-engine}` (le reste du monorepo garde la couverture de lignes ≥85 %, gate existante — SEC-005 ne l'étend pas).
- Augmentation de la couverture de LIGNES (ratchet ≥85 %, gate CI existante) — SEC-005 mesure la QUALITÉ des tests, pas leur quantité.
- Charge (SEC-004) · isolation (SEC-002) · audit (SEC-001) · PRA (SEC-003).
- **Blocage explicite** : ne pas dispatcher tant que REP-001 (F7) n'est pas DONE, sauf décision de scope réduit « moteur de file seul » consignée.
