# SEC-004 : Charge k6 — 100 tickets/min/agence × 50 agences, Socket.io p95 <500ms, tuning PgBouncer

**Module** : F9 — Sécurité renforcée & charge · **Agent** : direct (perf/infra) · **Dépend de** : RT-002 (DONE — protocole de mesure normé p95, adapter Redis multi-instance), F3 DONE, RT-001 (réel branché) · **Statut** : TODO

**Révision** : v1 — expansion Boucle 1. Test de charge k6 dimensionnant le système au débit cible réseau, sous adapter Redis multi-instance, avec introduction et **tuning de PgBouncer** (pooling de connexions). **Réutilise le protocole de latence normé de RT-002** (p95 loopback), qu'il étend à l'échelle 50 agences — ne le redéfinit pas.

> **Périmètre perf/infra.** Artefacts en `ops/load/` (scénarios k6, config PgBouncer, rapports de seuils) + job CI/planifié. Ne touche ni `apps/` ni `packages/`.

## Cibles chiffrées (verrouillées)
- **Débit** : **100 tickets/min/agence × 50 agences = 5 000 tickets/min ≈ 83,3 tickets/s** en régime soutenu (POST /public/tickets + cycle appel/service/clôture proportionnel).
- **Latence temps réel** : Socket.io `ticket:called` **p95 < 500 ms** sous charge, mesuré au **protocole normé RT-002** (`t0=io.emit` serveur, `t1=handler client`, ≥50 échantillons, warm-up exclu, transport websocket, **adapter Redis ACTIF**, multi-instance). En cas de flake CI → quarantaine/retry documenté, JAMAIS un gate horloge-murale par appel (leçon F3/RT-002).
- **Latence HTTP** : POST /public/tickets p95 < 800 ms et p99 < 1500 ms sous débit cible (seuils verrouillés, ajustables en config).
- **Taux d'erreur** : < 0,1 % de requêtes en erreur (hors 429 attendus du rate-limit) sur la durée du palier soutenu.
- **Pool de connexions** : sous 50 agences, le nombre de connexions PG directes reste borné par PgBouncer (mode transaction), sans épuisement du pool ni erreurs `too many connections`.

## Exigences (EARS)

- **UBIQUITAIRE** : un scénario k6 DOIT modéliser 50 agences émettant chacune 100 tickets/min (montée en charge → palier soutenu → descente), avec un mix réaliste (émission, call-next, serve, close, feedback) respectant le rate-limit public déjà en place (les 429 attendus sont comptés à part, pas comme erreurs).
- **QUAND** le palier soutenu (débit cible) est atteint, ALORS `ticket:called` p95 DOIT rester < 500 ms (protocole RT-002, adapter Redis actif multi-instance) ET POST /public/tickets p95 < 800 ms — assertés par les `thresholds` k6 (le job échoue si un seuil est franchi).
- **PgBouncer** : le système sous charge DOIT router les connexions PG via **PgBouncer en mode transaction** ; la config (pool size, `default_pool_size`, `max_client_conn`) DOIT être versionnée (`ops/load/pgbouncer.ini`) et **tunée** par itérations mesurées ; un rapport DOIT documenter le réglage retenu et la marge (connexions PG utilisées vs max).
- **SI** un seuil (`thresholds`) est franchi (p95 latence, taux d'erreur, épuisement pool), ALORS le run k6 DOIT échouer avec un rapport exploitable (percentiles, erreurs par type, saturation) — pas de « vert mou ».
- **Reproductibilité** : le run DOIT s'exécuter sur un environnement contrôlé documenté (specs, loopback ou réseau contrôlé, PG16+Redis+PgBouncer réels) ; les résultats bruts (JSON k6) sont archivés comme artefact CI.
- **anormal** : sous surcharge au-delà de la cible, le système DOIT **dégrader proprement** (rate-limit 429 + backpressure), JAMAIS s'effondrer (pas de cascade de 500, pas de deadlock DB) — un scénario de dépassement volontaire vérifie la dégradation contrôlée.
