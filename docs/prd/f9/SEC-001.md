## Modèle d'immuabilité (défini, pas ré-implémenté)
L'immuabilité est **au niveau base** (DB-004, DONE) : `audit_log` interdit UPDATE/DELETE par trigger `BEFORE UPDATE/DELETE ... RAISE EXCEPTION` + `REVOKE UPDATE, DELETE` au rôle applicatif `sigfa_app`. **Décision d'architecture consignée : PAS de hash chain applicative en F9.** Justification : (1) l'append-only est déjà garanti côté moteur SQL (le rôle applicatif ne peut PHYSIQUEMENT pas muter une ligne) ; (2) une hash chain protège contre un attaquant ayant l'accès `owner`/superuser PostgreSQL — menace couverte par la restriction d'accès infra + PRA (SEC-003), pas par l'application. Une hash chain (colonne `prev_hash`/`entry_hash` chaînée) est **listée en question ouverte** (`_notes.md`) pour une éventuelle exigence de conformité future (BCEAO/audit externe) mais **hors-scope de SEC-001**. L'immuabilité assertée ici = append-only vérifié par test (toute tentative UPDATE/DELETE → exception).

## Exigences (EARS) — SEC-001a (agent-api, branchement)

- **UBIQUITAIRE** : CHAQUE mutation applicative non-DDL qui n'est PAS déjà journalisée par un trigger DB-004 (rappel : `tickets` est INTENTIONNELLEMENT hors trigger — fréquence d'UPDATE) DOIT appeler `insertAuditEntry()` dans la MÊME transaction que la mutation, avec `{ actorId, actorRole, actorEmail, action, entityType, entityId, ip, diff }`. Le périmètre exhaustif couvre au minimum : transitions de ticket (`CALLED`, `SERVING`, `DONE`, `NO_SHOW`, `ABANDONED`, `TRANSFERRED` — API-003), appels/débordements/pause de file (API-004), sync offline (API-005), statuts agent & alertes de déconnexion→WAITING (API-007), feedback (API-010), révocation de session borne (API-009/011). Un **registre versionné** `apps/api/src/audit/mutation-registry.ts` énumère `{ route, method, entityType, action }` de TOUTE mutation de l'API.
- **QUAND** une mutation réussit (commit), une entrée `audit_log` correspondante DOIT exister ; **SI** la mutation échoue/rollback, ALORS AUCUNE entrée ne doit être écrite (atomicité — l'insert d'audit partage la transaction).
- **`action`** doit être une chaîne stable et lisible (`« PATCH /tickets/:id/close »`, `« POST /tickets »`) ; **`diff`** doit contenir `{ before, after }` en EXCLUANT toute colonne au motif `*_hash`/`*_encrypted`/`*_cipher` et le téléphone en clair (cohérent DB-004 + DB-008). Le téléphone n'apparaît JAMAIS en clair dans `diff`.
- **`ip`** DOIT provenir de l'IP client réelle résolue par le trust-proxy XFF déjà durci (F3), JAMAIS d'une IP fournie dans le payload ; **`occurred_at`** vient de la base (`DEFAULT now()`), jamais du client.
- **Test de couverture exhaustive (anti-régression)** : un test parcourt le `mutation-registry.ts` et, pour CHAQUE entrée, exécute la mutation puis assert qu'une ligne `audit_log` a été créée. Une mutation ajoutée sans entrée de registre OU sans audit fait ÉCHOUER ce test (le registre est la source de vérité, comparé aux routes montées).
- **anormal** : `insertAuditEntry()` ne doit JAMAIS avaler d'erreur silencieusement ; un échec d'écriture d'audit fait échouer (rollback) la mutation métier (l'audit est une pré-condition de commit, pas un best-effort).

## Exigences (EARS) — SEC-001b (agent-web, écran Auditor)

- **API-First** : l'écran consomme EXCLUSIVEMENT `GET /audit-logs?entityType=&entityId=&actorId=&from=&to=` (CONTRACT-005, schéma `AuditEntry { actor, action, entityType, entityId, timestamp, ip, diff }`) via le client typé généré. **Aucun fetch vers une route absente du contrat.**
- **LÀ OÙ** le rôle du claim JWT n'est pas `SUPER_ADMIN` ou `AUDITOR`, la route d'écran Auditor DOIT retourner 403 côté serveur (middleware layout RSC, cohérent WEB-001) — aucun composant client ne prend de décision d'accès.
- **UBIQUITAIRE — orthogonalité lecture seule (leçon SEC-F3-01)** : l'écran Auditor n'expose AUCUN bouton, champ ou appel de mutation. Le DOM ne contient aucun élément d'action (pas de POST/PATCH/DELETE émis). L'AUDITOR ne peut RIEN muter, y compris sur les logs eux-mêmes.
- L'écran doit offrir : filtres (type d'entité, entité, acteur, plage `from/to`), pagination conforme au contrat, affichage lisible du `diff` (`before`/`after`), horodatage en fuseau agence, et les 5 états d'écran (nominal / loading / empty « aucun événement » / error / offline).
- **i18n FR/EN uniquement** (décision PO — Dioula/Baoulé retirés) : libellés, en-têtes de colonnes et messages d'état traduits FR/EN. Tokens de design uniquement, jamais de valeur en dur.
- **Scope tenant** : un AUDITOR de banque A ne voit QUE les logs de banque A (le scope est appliqué serveur via RLS/`withTenant` — SEC-002 ; l'écran ne fait aucune hypothèse de filtrage client).
