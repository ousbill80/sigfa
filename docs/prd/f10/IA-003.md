# IA-003 : Détection anomalies — file bloquée, agent inactif, SLA systématique

**Module** : F10 — IA & Prédiction · **Agent** : agent-api · **Dépend de** : IA-001 (pipeline features) · **Statut** : PRD PRÊT — **implémentation GATED sur données réelles de production**

> **⚠️ Gating produit** : la détection de motifs récurrents (SLA systématique, patterns d'inactivité) exige un historique réel pour calibrer les seuils. **Implémentation runtime BLOQUÉE** sur données de production. **Backtest sur scénarios synthétiques** entièrement testable AVANT production (critères ⊛). Voir `_notes.md`.

> **Frontière contractuelle (CONTRACT-008)** : une **alerte** temps réel (`alert:manager`, CONTRACT-002 / API-007) est **instantanée** (1 occurrence). Une **anomalie IA** est un **motif agrégé** dans le temps. IA-003 ne remplace pas les alertes API-007 : elle **agrège** des signaux en anomalies persistantes acquittables. Types énumérés = **`QUEUE_STUCK`, `AGENT_INACTIVE_PATTERN`, `SLA_SYSTEMIC`** (CONTRACT-008, jamais d'autre valeur sans story contrat amont).

## Exigences (EARS)

- **UBIQUITAIRE — détection par tenant, sur signaux SIGFA** : la détection doit opérer **uniquement** sur les features/faits SIGFA de la banque (IA-001, alertes API-007 agrégées), isolée par `bank_id` (garde applicative worker, hors RLS de session).
- **QUAND (WHEN) `QUEUE_STUCK` est détecté** : le système doit lever une anomalie SI une file n'avance pas (aucun `ticket:called` net) pendant une fenêtre configurable (défaut **≥ 15 min** avec ≥ N tickets en attente et ≥ 1 guichet théoriquement ouvert), avec preuves (`agencyId`, `serviceId`, fenêtre, tickets bloqués).
- **QUAND (WHEN) `AGENT_INACTIVE_PATTERN` est détecté** : le système doit lever une anomalie selon le **seuil documenté CONTRACT-008 — ≥ 3 alertes `AGENT_INACTIVE` sur 7 jours glissants** pour le même agent (seuils surchargables via `/banks/:id/thresholds`, CONTRACT-005). C'est un **motif**, pas l'alerte instantanée d'API-007.
- **QUAND (WHEN) `SLA_SYSTEMIC` est détecté** : le système doit lever une anomalie SI le **taux SLA** (REP-001) d'un service/agence reste sous cible sur une fenêtre récurrente (défaut : sous cible ≥ 3 jours ouvrés sur 5 glissants), avec le taux mesuré et la cible.
- **UBIQUITAIRE — cycle de vie & idempotence** : une anomalie doit suivre `open → acked → resolved` (`POST /ai/anomalies/:id/ack`, CONTRACT-008). Le détecteur doit être **idempotent** : réexécuter sur la même fenêtre ne crée pas de doublon — une anomalie active pour la même clé `(bank_id, type, scope, window-key)` est **mise à jour**, pas dupliquée.
- **UBIQUITAIRE — explicabilité** : chaque anomalie doit porter des **preuves lisibles** (`evidence`) : métrique observée, seuil, fenêtre, échantillon de faits — pour qu'un manager comprenne **pourquoi** sans expertise data.
- **UBIQUITAIRE — pas d'action automatique** : une anomalie ne doit **jamais** déclencher d'action corrective automatique (réaffectation, ouverture guichet, sanction agent). Elle **signale** ; l'humain décide. Test explicite : aucune mutation opérationnelle émise par le détecteur.
- **SI (IF) l'historique / la fenêtre est insuffisant(e) pour un type ALORS** ce type de détection est **suspendu** (pas de faux positifs sur données trop maigres) et l'état est traçable ; les endpoints suivent la sémantique `INSUFFICIENT_HISTORY` de CONTRACT-008 le cas échéant.
- **ANORMAL (IF…THEN)** : SI un même signal alimente à la fois une alerte instantanée (API-007) et une anomalie IA ALORS il ne doit pas y avoir de double comptage — l'anomalie **référence** les alertes agrégées sans les recréer.
- **UBIQUITAIRE — métriques d'évaluation** : la qualité du détecteur doit être mesurée sur un **jeu de scénarios étiquetés** (backtest) : **précision / rappel / F1** par type d'anomalie, avec un objectif de **maîtrise des faux positifs** (le bruit détruit la confiance opérationnelle). Un scénario « nominal » ne doit produire **aucune** anomalie.
