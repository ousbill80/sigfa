# IA-001 : Pipeline features — historique affluence par agence/heure/jour + calendrier CI

**Module** : F10 — IA & Prédiction · **Agent** : agent-api · **Dépend de** : REP-001 (moteur d'agrégats), DB-007 (tables IA) · **Statut** : PRD PRÊT — **implémentation GATED sur données réelles de production**

> **⚠️ Gating produit (rappel catalogue F10)** : F10 a pour prérequis explicite « données réelles de production ». Cette story est **expansée et DONE-able au sens PRD**, mais son **implémentation runtime est BLOQUÉE** tant qu'aucune banque pilote n'a accumulé l'historique minimal (**seuil v1 : 90 jours** de tickets fermés, aligné CONTRACT-008 `INSUFFICIENT_HISTORY`). La partie **backtest / jeux synthétiques** est testable AVANT données réelles (voir critères ⊛). Voir `_notes.md`.

> Cette story ne produit **aucune prédiction** : elle construit uniquement le **pipeline de features** (extraction, calendrier, matérialisation) qui alimente IA-002/003/004. La couche modélisation est hors périmètre ici.

## Exigences (EARS)

- **UBIQUITAIRE — source unique = données SIGFA du tenant** : le pipeline doit extraire les features **exclusivement** depuis les agrégats/faits SIGFA de la banque (tickets fermés, `waitTime`, `serviceTime`, abandons, occupation guichets — sortie REP-001) et depuis les tables IA (DB-007). Aucune donnée externe au tenant, aucune donnée d'une autre banque (isolation `bank_id`). Le `bank_id` porté par chaque feature est LA source de vérité (les jobs tournent hors requête HTTP, donc hors RLS de session — garde applicative obligatoire, cf. NOTIF-001).
- **UBIQUITAIRE — granularité canonique** : chaque enregistrement de feature doit être clé par `(bank_id, agency_id, service_id?, date, hour_bucket)`. Le `hour_bucket` par défaut est **horaire** (0–23) ; le pipeline doit rester paramétrable au **créneau de 30 min** sans changement de schéma (colonne `bucket_minutes ∈ {30, 60}`, défaut 60).
- **UBIQUITAIRE — features exactes v1** : pour chaque bucket, le pipeline doit calculer :
  - `arrivals` (tickets émis), `served`, `no_show`, `abandoned`
  - `avg_wait_s`, `p90_wait_s`, `avg_service_s` (réutilisent les calculs certifiés REP-001, jamais recalculés à la main)
  - `counters_open` (nb guichets ouverts observés), `agents_active`
  - features **calendaires** (ci-dessous), features **lag** : `arrivals_lag_7d` (même bucket J-7), `arrivals_lag_1d`, `arrivals_roll_mean_4w` (moyenne du même bucket sur 4 semaines glissantes).
- **UBIQUITAIRE — calendrier Côte d'Ivoire** : le pipeline doit joindre un **calendrier CI** fournissant, par date : `is_month_end` (3 derniers jours ouvrés du mois), `is_public_pay_day` (fenêtre paie fonction publique — **paramétrable par banque**, défaut : 25 au dernier jour du mois), `is_public_holiday` (jours fériés CI, source = seed DB-003), `day_of_week`, `is_eve_of_holiday`. Les libellés de facteurs doivent correspondre **exactement** à l'énumération contractuelle CONTRACT-008 (`fin de mois`, `paie fonction publique`, `férié`).
- **QUAND (WHEN) le job de matérialisation s'exécute** (planifié, ex. nocturne + rattrapage), le système doit calculer/mettre à jour les features des buckets de la fenêtre demandée de façon **idempotente** : réexécuter sur la même fenêtre produit exactement les mêmes lignes (upsert par clé, jamais de doublon).
- **QUAND (WHEN) une plage de dates est incomplète** (agence créée récemment, jour partiel, coupure), le pipeline doit marquer les buckets concernés `is_partial = true` et **ne jamais fabriquer** de valeurs (pas d'imputation silencieuse) — les consommateurs (IA-002/003) décident du traitement.
- **SI (IF) l'historique disponible d'une agence < seuil (défaut 90 jours de tickets fermés) ALORS** le pipeline doit produire les features disponibles mais exposer un indicateur `available_days` par agence, réutilisé tel quel par les endpoints IA pour renvoyer `422 INSUFFICIENT_HISTORY` (CONTRACT-008) — le pipeline **n'invente pas** d'historique.
- **SI (IF) une correction rétroactive survient** (ticket ré-agrégé par REP-001, jour ré-ouvert) **ALORS** un re-run du pipeline sur la fenêtre impactée doit reconverger vers les valeurs corrigées (traçabilité `computed_at`, `source_watermark`).
- **UBIQUITAIRE — rétention & versionnage** : la matérialisation doit respecter la rétention DB-007 et estampiller `feature_set_version` (schéma des features) pour que IA-002/003/004 sachent sur quelle version elles ont été entraînées/évaluées.
- **UBIQUITAIRE — reproductibilité / backtest** : le pipeline doit accepter une **fenêtre historique bornée** (`from`, `to`) et produire un **jeu de features déterministe** exploitable en backtest hors production, à partir d'un **jeu de données synthétique** conforme au schéma REP-001/DB-007.

## Critères d'acceptation

- [ ] `IA-001: features clés (bank_id, agency_id, service_id?, date, hour_bucket) uniques — upsert idempotent, re-run même fenêtre = zéro doublon (test intégration Testcontainers)` ⊛
- [ ] `IA-001: features v1 calculées à partir des sorties REP-001 (aucun recalcul TMA/TMT ad hoc) — valeurs identiques au moteur d'agrégats sur dataset de référence (test)` ⊛
- [ ] `IA-001: calendrier CI — is_month_end / is_public_pay_day / is_public_holiday / is_eve_of_holiday corrects sur une année de dates connues, fériés = seed DB-003 (test déterministe)` ⊛
- [ ] `IA-001: libellés de facteurs = énumération CONTRACT-008 exacte (fin de mois, paie fonction publique, férié) (test structurel)` ⊛
- [ ] `IA-001: features lag (lag_7d, lag_1d, roll_mean_4w) exactes sur série synthétique connue (test)` ⊛
- [ ] `IA-001: buckets incomplets → is_partial=true, aucune imputation (test)` ⊛
- [ ] `IA-001: available_days par agence exposé et cohérent avec le seuil 90j (test)` ⊛
- [ ] `IA-001: isolation tenant — un job bank_id=A ne lit/écrit jamais de features bank_id=B (test tenant-isolation worker)` ⊛
- [ ] `IA-001: paramétrage bucket 30/60 min sans migration de schéma (test)` ⊛
- [ ] `IA-001: backtest — fenêtre (from,to) sur dataset synthétique produit un feature-set déterministe et rejouable (test)` ⊛
- [ ] `IA-001: [GATED données réelles] premier run sur historique pilote ≥90j produit des features non vides et cohérentes (validation pilote — hors CI)`

> ⊛ = **évaluable AVANT données réelles** (jeux synthétiques / backtest). Seul le dernier critère est gated sur l'historique de production.
