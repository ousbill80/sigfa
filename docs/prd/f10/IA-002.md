# IA-002 : Prédiction affluence + recommandation staffing

**Module** : F10 — IA & Prédiction · **Agent** : agent-api · **Dépend de** : IA-001 (pipeline features) · **Statut** : PRD PRÊT — **implémentation GATED sur données réelles de production**

> **⚠️ Gating produit** : la prédiction n'a de valeur qu'avec un historique réel. **Implémentation runtime BLOQUÉE** tant que l'agence pilote n'atteint pas le seuil (défaut **90 jours** de tickets fermés, CONTRACT-008). Tant que l'historique est insuffisant, `GET /ai/forecast` et `GET /ai/staffing-recommendations` renvoient **`422 INSUFFICIENT_HISTORY`** (comportement déjà contractualisé). L'**évaluation par backtest sur jeux synthétiques** est testable AVANT production (critères ⊛). Voir `_notes.md`.

> **Garde-fou cardinal (rappel constitution)** : une recommandation staffing est une **aide à la décision**. Le système ne doit **jamais** ouvrir/fermer un guichet ni réaffecter un agent automatiquement. Toute action reste **humaine** (acquittement explicite).

## Exigences (EARS)

- **UBIQUITAIRE — modèle par tenant** : le modèle de prédiction doit être entraîné **uniquement** sur les features SIGFA de la banque concernée (IA-001), jamais sur des données d'un autre tenant. Chaque prédiction porte `AiMeta { modelVersion, computedAt, dataWindow }` (CONTRACT-008).
- **QUAND (WHEN) `GET /ai/forecast?agencyId=&date=` est appelé avec historique suffisant**, le système doit retourner une **série horaire** `{ hour, expectedTickets, confidence }` pour la date demandée (± horizon court, défaut J..J+7), plus les **facteurs contextuels** ayant pesé (`fin de mois`, `paie fonction publique`, `férié` — énum CONTRACT-008).
- **UBIQUITAIRE — explicabilité** : chaque point de forecast doit être **explicable** : le système doit fournir, pour la journée et/ou le pic, les **top facteurs contributifs** (ex. « +40 % vs médiane : paie fonction publique + fin de mois ») dans un champ `drivers[]` structuré (facteur, sens, poids relatif). Aucune recommandation « boîte noire ».
- **QUAND (WHEN) `GET /ai/staffing-recommendations?agencyId=&date=` est appelé**, le système doit dériver de la prédiction une liste `{ time, action, counters, rationale }` (ex. « ouvrir 2 guichets à 10h30 »), où `rationale` cite explicitement la prédiction et l'objectif SLA visé (réutilise le taux SLA REP-001). Chaque recommandation est **acquittable** via `POST /ai/staffing-recommendations/:id/ack` (statut open/acked).
- **UBIQUITAIRE — pas de décision automatique** : le système ne doit **jamais** déclencher d'action opérationnelle (ouverture guichet, affectation, notification client) à partir d'une recommandation. Il **propose**, un humain (DIRECTOR+) **décide et acquitte**. Test explicite : aucune mutation d'état guichet/agent n'est émise par le moteur IA.
- **SI (IF) l'historique de l'agence < seuil (défaut 90 j) ALORS** `/ai/forecast` et `/ai/staffing-recommendations` renvoient `422 INSUFFICIENT_HISTORY { requiredDays: 90, availableDays }` (CONTRACT-008), sans jamais fabriquer une prédiction non fondée.
- **SI (IF) la confiance d'un point est sous un seuil configurable ALORS** le point doit être marqué `lowConfidence: true` et **aucune** recommandation staffing ne doit en être dérivée seule (garde anti-sur-réaction).
- **ANORMAL (IF…THEN)** : SI le pipeline features est en retard / partiel (`is_partial`, cf. IA-001) sur la fenêtre demandée ALORS la réponse doit signaler la **fraîcheur dégradée** (`dataWindow` reflète l'étendue réelle) plutôt que d'extrapoler silencieusement.
- **UBIQUITAIRE — métriques d'évaluation** : la qualité du modèle doit être mesurée et journalisée par un **backtest** : **MAE** et **MAPE** sur `expectedTickets` par bucket, plus **calibration** de `confidence` (couverture de l'intervalle). Un **garde-fou de non-régression** compare le modèle candidat à une **baseline naïve** (moyenne du même bucket J-7 / 4 semaines, dérivée de `arrivals_lag_7d`/`roll_mean_4w` d'IA-001) : le modèle candidat doit **battre la baseline** sur le jeu de backtest, sinon il n'est pas promu.

Voir `_notes.md`. Aucune route hors CONTRACT-008.

Aucune décision opérationnelle automatique (l'humain tranche). Aucun connecteur Core Banking / CRM / Mobile Money / USSD / biométrie / BCEAO. Aucun modèle cross-tenant. Aucune PII brute en entrée du modèle (features agrégées uniquement, IA-001). SIGFA reste 100 % standalone.
